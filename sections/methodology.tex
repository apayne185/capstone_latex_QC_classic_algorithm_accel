\section{Methodology}
\label{sec:methodology}

This proposed triple-heterogeneous software stack follows a high performance middleware design pattern, specifically
focusing is structure on the orchestration of distributed classical resources alongside remote, cloud based quantum backends, we can 
address the preestablished "Communication Wall" \cite{weder2021quantum} through the development of an asynchronous and 
parallelized pipeline that executes the hybrid quantum-classical VQE software to find the ground states energies of benchmark molecules $H_2, LiH, BeH_{2}, H_{2}O$  \cite{kandala2017hardware}
with a speedup of 2-3x as compared to a classical counterpart. 

\subsection{Distributed System Architecture}
This stack follows a modular architecture that is designed to decouple the classical, high performance subroutines of SPSA optimization, Pauli term evaluation, local statevector
construction, and the distributed MPI coordination from the quantum circuit execution of circuits, and the measurement of quantum expectation values in order to allow for asynchronous execution. 
There are three distinct layers defined within the architecture: the Orchestration layer, the Acceleration layer, and the Quantum interface layer. This system is designed specifically to handle the varying timescales of different components, including 
GPU accelerated classical math (microseconds), MPI internode communication (milliseconds), and the cloud based 
QPU Round-Trip Time (RTT)(seconds). Through these considerations, this middleware stack moves beyond the previously examined single host, synchronous execution model found in the 
examples of hybrid middleware like XACC \cite{mccaskey2018xacc} and the HWE-SPSA-VQE research-based setup of Kandala et al. \cite{kandala2017hardware} that was explored in this paper's Literature Review.

\subsubsection{Orchestration Layer: Distributed Memory and MPI Manager-Worker Topology}
This top-level layer implements a manager-worker topology that utilizes Message Passing Interface (MPI) in order 
to manage a cluster of compute nodes (workers). Throughout this process, the manager node (rank 0) handles the global 
state of the classical optimizer SPSA, holding the current $\theta$ parameters, determining the values of $\theta^+$ and $\theta^-$, and distributing the 
variational parameter sets $\theta$ across the ranks of available classical workers to enable concurrent gradient 
estimation. SPSA is a derivative-free optimizer, found well suited to handle the quantum measurement noise found on 
physical qubits (unlike the gradient based methods of Adam or SGD) \cite{pellow2021classicoptimizers}. Due to 
this, SPSA is recommended choice for VQE (and for our NISQ hardware) due to its ability to handle the physically noisy 
objective functions. 
    
By utilizing \texttt{MPI\_Isend} and \texttt{MPI\_Irecv} for non-blocking asynchronous communication between the 
classical-quantum components, the stack allows nodes to remain active and begin the local computations of the state vector expectation values while still
waiting for the circuit results from the cloud-based QPU, addressing the identified Communication Wall \cite{weder2021quantum}. Once worker nodes receive the expectation values, they 
execute a \texttt{MPI\_allreduce} operation to aggregate the results of each ranks partial energy and then to update the global objective function.

\subsubsection{Acceleration Layer: Intra-Node CUDA-Kernel Execution}
This layer follows the findings of Bayraktar et al. \cite{bayraktar2023cuquantum} and is designed in order to 
offload the computationally heavy $2^n$ state-vector math to each MPI rank's NVIDIA GPU. The stack implements a three
tiered acceleration strategy, picking the best option between two accelerated classical processing and the automatic local fallback in place:

\begin{enumerate}
    \item \textbf{GPU statevector (cuStateVec)}: Best option, used when an NVIDIA GPU is detected. The stack switches to Qiskit Aer's GPU-accelerated statevector simulator backed by NVIDIA's cuStateVec library from the cuQuantum SDK. This accelerates the $O(2^n)$ local statevector construction, which dominates per-iteration cost \cite{bayraktar2023cuquantum}, along with all gate operations (CNOT, rotations) being executed as GPU matrix multiplications.
    \item \textbf{CUDA Pauli kernel}: A custom CUDA kernel handles the Pauli term expectation values using a mean-field approximation with one GPU thread per 15-1086 Pauli term, where then partial results are then summed in tree reduction in shared memory. This kernel processes the distributed Pauli terms acroos the MPI ranks.
    \item \textbf{CPU fallback}: When no GPU is available, the stack falls back to Qiskit's CPU-based Python/C++ \texttt{Statevector} class for exact (noiseless) state-vector simulation. 
\end{enumerate}

The CUDA kernel adopts a Mixed Precision strategy using FP32/FP64 (32, 64 bit floating-point), used to balance the stacks need 
for high computational throughput with the need for numerical accuracy. FP32 (Single Precision) is faster but less precise, used to handle the trigonometric computations 
of \texttt{cosf} and \texttt{sinf} within each Pauli term evaluation (which are intermediate values that need only 7 decimal digits of precision), which provides this stack up to 2x faster throughput 
on the GPU cores. FP64 (Double Precision), the slower but exact option, is used in the tree reduction (where all Pauli term contributions are saved) and global accumulation phases in 
order to prevent precision loss (rounding errors) that can occur during the summation of many small floating point numbers. The MPI \texttt{Allreduce} 
aggregation operates within FP64, ensuring that the final energy estimate will maintain a full double precision accuracy across all MPI ranks. These strategies were combined to reduce the memory bandwith bottleneck
that Bayraktar et al. recognized within state vector simulation, where the issue stems from reading/writing the statevectors from GPU memory instead of the floating point computation \cite{bayraktar2023cuquantum}. Considering FP32 values are 4 bytes while
FP64 are 8 bytes, using FP32 wihtin per-term evaluations will cut the memory bandwidth needed in these operations in half, doubling the bandwidth effective throughput on this hardware. Within the chosen GPU on this stack, this will prove critical. FP32 
also assist in maintaining chemical accuracy within our threshold of $\tau = 1.6$mHa, as it provides 7 decimal digits of precision within our distributed task of trig operations in a Pauli term evaluation. FP64 is brought in to address the
precision risks within accumulation (consider summing 1086 small values for the molecule $H_{2}O$). This ensures we maintain chemical accuracy and still maximize the throughput. 

\subsubsection{Quantum Interface Layer - Asynchronous Dispatcher}
This layer is the bottom of the stack, conducting the abstraction of how quantum hardware-specific drivers (or simulators) perform computations, allowing the previous layers to function regardless if they are using a local simulator (CPU), local GPU, or a cloud GPU. This extends the core ideas of XACC's accelerator model \cite{mccaskey2018xacc}.   
This layer handles job submission to all options of local simulators and cloud based QPU backends, routing to either a GPU kernel or a CPU fallback based on the detected hardware. Two dispatch paths occur:

\begin{enumerate}
    \item \textbf{C++ dispatcher} (\texttt{dispatcher.cpp}): Manages the MPI synchronization with rank 0 using \texttt{MPI\_Bcast} for $\theta$  parameter distribution and \texttt{MPI\_Allreduce} for the partial energy aggregation from all ranks, the sum to be later broadcasted. Routes computation to either the CUDA Pauli kernel (when NVIDIA GPU is available) or to the CPU mean-field fallback path. For the cloud IBM QPU path, \texttt{std::async} is used for nonblocking job submission through a launched REST client on a background thread ensures the main thread does not block, implementing this stacks latency hiding strategy. (\texttt{qpu\_client.cpp}) handles IBM Quantum's IAM authentication, JSON payload construction, and polling with exponential backoff (waiting 1s,2s,4s,8s). 

    \item \textbf{Python EstimatorV2 path} (\texttt{interface.py}): For IBM Quantum experiments, the stack uses Qiskit's EstimatorV2 primitive, which provides the stack with simpler built-in transpilation, observable layout mapping, and with T-REx measurement error mitigation for NISQ hardware \cite{cai2023quantumerrormitigation}, instead of the C++ path's REST calls. As SPSA needs both $E(\theta^+)$ and $E(\theta^-)$ gradient estimations per iteration \cite{pellow2021classicoptimizers}, they get bundled into one job as two Primitive Unified Blocs (PUBs), therefore reducing the number of QPU round-trips by 50\%. During the QPU wait time, all MPI ranks independely compute exact statevector expectation values, providing this path both a noise-free reference estimate and useful classical work that contributes to the masking metric $M$ \cite{weder2021quantum}.
\end{enumerate}

This dual path design allows the stack to determine and then utilize the most appropriate interface for each backend, 
chosing either the high level Python path for cloud QPU access (with its built-in error mitigation strategy), or the 
low level C++ dispatcher to better handle high-throughput local computation with GPU acceleration.

\subsection{Algorithmic Workflow: Parallelized VQE}
The Variational Quantum Eigensolver algorithm is reformulated in this stack from Peruzzo et al.'s single host VQE\cite{peruzzo2014variational}, in order to exploit its distributed nature, 
following an iterative cycle of: 

    \begin{enumerate}
        \item Global Parameter Broadcast: The optimizer rank broadcasts a vector of parameters ($\theta$) to $P$ parallel nodes.

        \item Parallel Expectation Value Estimation: Each node ($p \in P$) compute the subset of the Pauli string expectation values: $$\langle\psi(\theta)|\hat{O}_i|\psi(\theta)\rangle$$. This is the stacks primary point of SIMD parallelization \cite{bayraktar2023cuquantum}, $O_i$ terms are distributed across the HPC cluster.

        \item Asynchronous Handshake: When using the cloud QPU backend, circuits queue on the remote processor while the classical MPI ranks will independently compute the exact statevector expectation values locally. This overlap provides both noise-free reference estimate and classical work that masks the QPU round-trip latencies\cite{weder2021quantum}.

        \item Reduction and Updating: Partial expectation values from all ranks get aggregated using \texttt{MPI\_Allreduce} into a single energy sum, which the classical gradient-free optimizer SPSA \cite{pellow2021classicoptimizers} uses (alongside the pertubed $E(\theta^+)$ and $E(\theta^-)$) to computes the next $\theta$ for the following iteration.
    \end{enumerate}

\subsection{Performance Evaluation and Experimental Design}
This section describes the full evaluation of this middlware stack, which will be validated through a Strong Scaling analysis where the  
stack's performance when total problem size ($W$ = size of workload) is kept fixed while the number of cores is increased ($P$ = number of processors) \cite{cuboulderScaling} is measured to determine points of speedup. This method of 
analysis is crucial to our stack, as it excels with reducing execution time with systems that focus on temporal limitations and the potential of parallelizing a systems workload. This measurement is governed by Amdahl's Law \cite{amdahl1967validity}, 
a formula establishing the upper limit of a speedup to be determined by the serial, non-parallelized portion of the problem, used to calculate the maximum (theoretical) speedup of a parallel system. Amdahl asserts that a threshold exists where adding processors will provide diminishing returns instead of computational speedups. 
While strong scaling measures the parallelization efficiency in a fixed environment, a Weak Scaling analysis is also implemented 
to test the middleware's communication overhead to determine if it grows sublinearly as the system scales. Weak scaling excels in determining if increasing both $W$ and $P$ will allow developers to solve larger problems within a consisent 
time-to-solution despite the growing problem size. In the work of Gustafson \cite{gustafson1988reevaluating}, he reasserts that massively parallel systems can still operate in a highly efficient manner to tackle 
a problem size that scales to the size of the massive system. Both analysises will be conducted to test the ability of this middleware to scale both in its parallelization efficiency and within the task of handling larger molecular systems. 

\subsubsection{Experimental Configuration}  
All simulator experiments of this stack were conducted inside a Docker container (CUDA toolkit 12.6.3, Qiskit 2.1+, qiskit-ibm-runtime 0.45.1, Python 3.11, PySCF 2.4+, OpenMPI from Ubuntu 22.04 system package) 
on a single host machine (a standard laptop setup, reflecting modern day hybrid quantum communication) Intel Core i7-1065G7 (Ice Lake, with 4 physical cores, 8 logical threads) clocked at 1.30 GHz base, 3.90 GHz turbo with 32 GB RAM, 
running on an Linux Mint OS. This configuration is set to ensure reproducibility across computing devices, along with proving the viability and portability of efficient hybrid computation on a commodity hardware device. 

This middleware utilizes a Variational Quantum Eigensolver (VQE) algorithm, configured with a HWE ansatz to ensure rapid computations on hardware adaptive, shallow circuits, along with the classical optimizer SPSA, 
effective for navigating quantum noise found on our NISQ devices. This setup follows a similar one to Kandala et al.'s, but expands to integrate dispatching to a GPU accelerated nodes for an enhanced computational speedup.  
SPSA has a set fixed pertubation size $c = 0.1$ that can appropriately handle varational 
angles within $[0,2\pi]$ with a step size of $a=0.628 / \sqrt{p/8}$, which scales inversely with the square root of the 
number of variational parameters. The stability constant, $A = 0.1 \times \text{max\_iterations}$ delays any aggressively early 
updates while the decay exponents of $\alpha= 0.602$ and $\gamma = 0.101$ follow the standard schedule of SPSA. The per-iteration update rule is:

\begin{equation}
    a_k = \frac{a}{(k + A)^\alpha}, \quad c_k = \frac{c}{k^\gamma}
\end{equation}

Convergence is determined by a sliding window of 10 iterations and the SPSA optimizer terminates when $\max(E) - \min(E) < \tau$ occurs 
within that window (as $\tau = 1.6$ mHa, the chemical accuracy threshold defined in this stack). An enforced minimum iteration count of $\max(10, \min(2p, 50))$ 
is applied to prevent false convergences from occuring on the small, early steps. A maximum iteration count scales alongside 
the number of parameters with $\max(200, 8p)$. For reproducibility, all experiments use a fixed random seed of 42. 

The ansatz configuration that is implemented within the stack is a tiered ansatz system that selects based on a correlation 
score metric. All four of the benchmark molecules fall within the tier of \texttt{hwe\_adaptive} tier (within correlation scores 0.34--0.50), 
using the EfficientSU2 circuit with full entanglement and $\min(\text{reps}+1, 3)$ number of repetition layers. The variational 
parameters use a uniform distribution during initialization ($\mathcal{U}(-0.1, 0.1)$), in order to keep the initial HWE 
state close to the Hatree-Fock reference $|0\ldots0\rangle$ under the Jordan-Wigner mapping to delay the occurance of particle number violations.  

All quantum backend experiments utilize IBM Quantum’s open access plan, which provides free access the 156 qubit \texttt{ibm\_marrakesh} production-scale processor without a financial barrier for research. IBM Quantum was selected as this stack's cloud QPU provider due 
to their Qiskit library providing the means for effective hybrid middleware integration, offering the EstimatorV2 primitive with built-in source code language conversion and error mitigation \cite{rallis2025interfacingquantumcomputingsystems}.  
In this stack, the Qiskit IBM Runtime EstimatorV2 primitive is configured with 4096 shots per circuit and with a resilience level 1 (T-REx measurement error mitigation) in order to balance the precision $1/\sqrt{4096} \approx 0.016\;E_h$ per expectation value while operating within the time limits of the IBM open access plan. 
The GPU-accelerated nodes are conducted on an NVIDIA GeForce GTX 1650 Mobile (Turing architecture, 4 GB GDDR6, CUDA 12.6), 
using both the Qiskit Aer CUDA thrust backend and the NVIDIA cuStateVec backend to accelerate the construction of the state vector and the local GPU gate operations. This consumer-grade GPU represents an accessible, lower bound 
in the range of performance-accelerator GPUs, whereas the less user accessible A100s (the GPU of choice benchmarked by Bayraktar et al.~\cite{bayraktar2023cuquantum}), if used 
would provide us with substantially improved speedups due to their higher memory bandwidth. However, this project largely relies on obtainable, cost-friendly resources, selecting the remote IBM Quantum open access plan and Lambda's NVIDIA GeForce GTX 1650 Mobile GPUs in order to develop and test this middleware. In future work, this project
will extend to computationally superior resources. 

\subsubsection{Application Scenarios}
To demonstrate the effectiveness of this middleware when tasked a scientifically meaningful workload, this project focuses on testing various real-world applications
quantum chemistry and computing the molecular electronic ground-state energies of four increasingly complex molecules over the iterations of the VQE. Chemistry was selected to be the  
application for this middleware due three main reasons: 
    \begin{enumerate}
        \item Molecular Hamiltonians (mathmatical description of electron interactions) will map naturally to a qubit operator ($X$,$Y$,$Z$,$I$) through the Jordan-Wigner transformation \cite{peruzzo2014variational}, which produces a clearly defined set of Pauli terms (ranging from 15 for $H_2$ to 1086 for $H_{2}O$) to then be batched and distributed across MPI ranks (while still not exceeding the coherency limitations of modern day QPUs). If this stack chose to operate beyond chemistry into fields such as finance or logistics, the decomposition into distributed subsets is not as simple. 
        \item The problem size will scaling predictably with the number of electrons and basis functions to allow for controlled experiments (ranging from 4-14 qubits as we scale from $H_2 - H_{2}O$). This alongs use to test the stack with Strong/Weak scaling analysises.  
        \item Lastly, it was selected due to the prexisting Full Configuration Interaction (FCI) energies which provide us a comparison to a verified, exact ground-truth references for each molecule (each molecules $E_0$) \cite{szabo1996modernquantumchemistry}, in order to understand the stacks ability to maintain an approximiate chemical accuracy while prioritizing a computational speedup. 
    \end{enumerate}

From the STO-3G minimal basis set (which provides us the qubit counts for representing each molecules orbitals) \cite{szabo1996modernquantumchemistry}, we selected four molecules of increasing complexity to be benchmark targets for 
this middleware stack. As a set, these molecules progressively provide us a sufficiently complex Hilbert space that can test the 
efficiency of the MPI/CUDA distribution while still using a manageable size of molecules that will still not exceed the coherency limitations of modern day QPUs.

\subsubsection{Scaling Strategies}
The objective of Strong Scaling is to reduce the execution time within a fixed total problem size by increasing the 
number of processors, while considering the ideal time spent on $N$ cores and efficiency \cite{amdahl1967validity}. For instance, this includes 
keeping the application scenario of $H_{2}O$, which has 1086 Pauli terms and needs 896 iterations, fixed while increasing the number of MPI nodes ($P \in \{1,2,4,8\}$). This would prove the functionality of parallelism in this stack, as if it is perfectly
parallel doubling the processors would cut the execution time in two. However, in real life, Amdahl's law is proven as the tasks of statevector construction and the communication overhead from MPI Allreduce will provide us with diminishing returns as scale increases. 
Ideal Time is the predicted time if parallelization was perfect (if Amdahl's law is proven incorrect), computed with the formula ($N$= number of nodes \cite{cuboulderScaling}): 
\begin{equation}
    \text {Ideal Time } = \frac{(\text{Measured Time at Lowest Core Count})\text{(Lowest Core Count)}}{N}
\end{equation}

Efficiency is the metric for the real-world application's appropriate number of cores for its particular problem size, informing us of how close the system comes to achieving Ideal Time. Achieving 100\% (the system's Ideal Time) is 
rarely possible within practice, so a there is a tradeoff point established between the Efficiency of the usage of the systems nodes
and the time spent waiting on the calculations to completed. The formula for Efficiency is \cite{cuboulderScaling}: 
\begin{equation}
    \text {Efficiency } = \frac{\text{Ideal Time}}{\text{Measured Time}}
\end{equation}
Within the implemented Strong Scaling in this stack, total wall-clock time ($T_{\text{total}}$), Efficiency, Ideal Time, and Parallel Efficiency ($E= T_1 / (P \times T_P)$) are measured. 

In addition to Strong Scaling, Weak Scaling is evaluated in this stack, where the scale of the chosen molecule (by it's Hamiltonian's Pauli term count) is 
increased proportionally alongside the number of MPI nodes ($P \in \{1,2,4,8\}$) on a single host \cite{gustafson1988reevaluating}. At each rank, a molecule from our set is selected that scales in size alongsize the number of ranks, however, due to the molecules chosen in the benchmarking, 
the term to rank scaling is not precisely constant, with $H_{2}$ is used at $P=1$ (15 terms/rank), then the terms scaling up considerably with $LiH$ at $P=2$ (316 terms/rank), $BeH_{2}$ at $P=4$ (167 terms/rank), 
and $H_{2}O$ at $P=8$ (136 terms/rank). The $P \geq 2$ ranks maintain a more consistent per-rank workload, while $P=1$ serves as the baseline example with its minimal complexity reference of $H_2$.  
This analysis will test whether the middleware maintains a stable compute-to-communication ratio on a single host, observing while the inputted problems scales up 
towards the Communication Wall\cite{weder2021quantum}. Here, the metrics of total wall-clock time (identical to Strong scaling), per-iteration time ($T_{\text{iter}}$), per-iteration MPI communication time ($T_{\text{comm}}$/iter)), and the median masking metric ($M$) are measured. 

Both scaling experiments were repeated with GPU acceleration so to compare against the CPU-only scaling results. 

\subsubsection{Stress Test Design}
To ensure an asynchronous dispatcher that can withstand obstacles without crashing and a functioning manager-worker topology, 
the system will undergo evaluations using a simulated "Failure Mode" that replicates real-world issues : 
    \begin{itemize}
        \item QPU Latency Spiking: A rnadom variable delay will be inserted (between 0.5--2s) into a subset of the worker-rank network calls, done to simulate increased and unpredictable cloud backend latency from QPU queues. This will test the manager ranks ability to rebalance workloads or to be able to maintain a global synchronizations using \texttt{MPI\_Allreduce} without rank timeouts or deadlocks. 
        \item Network Drop-Out Recovery: During active job submission (simulating a crash mid-run), we test the system by deleting the most recent saved checkpoint to test if the Orchestration Layer can perform a "checkpoint restart", in where the last know global $\theta$ state (the previously saved checkpoint before the deleted one) that is stored in the manager rank is used to resume optimization without a full system crash or requiring a system reboot. Each 5 iterations, the system automatically saves a checkpoint within checkpoints/ to provide the restart points. 
    \end{itemize}

\subsubsection{Baseline Implementations}
These are the control experiments of this stack, developed in order to verify that triple heterogeneous stack has an 
optimized performance, measured against two baselines: 
     \begin{itemize}
         \item Serial Baseline: Execution using the same Qiskit Statevector VQE on a single core CPU, using identical SPSA hyperparameters, the HWE-adaptive ansatz, and the same convergence criteria as the distributed MPI/GPU stack, but without any parallelization or GPUs. This identifies the outcome of the stack's MPI distribution from any VQE differences, providing a fair comparison of the wall-clock times between this nonoptimized industry standard starting point to a parallel, distributed system.
         \item Single Node GPU Baseline: Execution using Qiskit Aer with a CUDA thrust backend on a single NVIDIA GTX 1650 Mobile GPU ($P=1$) without the MPI distribution. This baseline is done to isolate the speedup contribution from GPU acceleration vs from the MPI orchestration, informing us from where in our stack that computational speedups are coming from and if the integration of GPUs/MPI is strictly necessary to see improvements in the stacks runtime.
     \end{itemize}
 
\subsubsection{Quantitative Metrics}
In the development of this stack, there are several metrics established that determine the success of this stack and the thresholds of how we determine
a pass/fail for different sections of the stack. The primary metric for determining success is the observed End-to-End Wall-Clock Time ($T_{total}$). 
   \begin{equation}
    T_{\text{total}} = T_{\text{accel}} + T_{\text{comm}} + T_{\text{quant}}
\end{equation}
Where: 
\begin{itemize}
    \item $T_{accel}$ = time spent in GPU accelerated optimization (classical acceleration).
    \item $T_{comm}$ = overhead from network RTT and MPI synchronization (communication latency).
    \item $T_{quant}$ = QPU execution time (sampling and measurement).
\end{itemize}

\noindent If on the simulator path, $T_{quant} = 0$, as there is no QPU use. If the quantity of metric $T_{comm}$ is successfully "masked" by the value of the asynchronous $T_{accel}$ 
execution time, there will be evidence of this stack's contribution towards assisting the "quantum advantage"\cite{harrow2017quantumsupremacy}, ensuring wait times and classical overhead will not undo a QPU speedup, along with the achievement of the targeted 2-3x speedup over standard single node, CPU-based Qiskit workflows.   

\noindent Masking Efficiency ($M$) will be used as a metric for how well the QPU RTT is hidden (masked) by the classical workload. 
\begin{equation}
    M= \frac{T_{\text{accel}}}{T_{\text{comm}}}
\end{equation}
If $M \ge 1$, then there is enough meaningful work within the classical components to completely cover (mask) the QPU communication wait time. 

\noindent To measure the stack's ability to handle high volume iterative workloads, Throughput (iterations per second) is measured as:
\begin{equation}
        \text{Throughput} = \frac{N_{\text{iterations}}}{T_{\text{total}}}
\end{equation}

\subsubsection{Correctness and Convergence}
For the purpose of ensuring the methods of optimization (such as Mixed Precision (FP32/FP64) and SPSA) do not compromise the chemical accuracy of 
results retuned by this stack, the tested molecules ground state energy results will be compared against the preestablished 
FCI results of the exact molecular ground state energies\cite{szabo1996modernquantumchemistry}. This software stack is considered successful 
within the application of quantum chemistry if it maintains Chemical Accuracy ($\approx 1.6$ mHa)\cite{szabo1996modernquantumchemistry} throughout the parallelized iterations. 

For the preparation of future, broader applications of this software, accuracy is also measured by Relative Error ($\epsilon$) : 
\begin{equation}
    \epsilon = \frac{|E_{exact} - E_{stack}|}{|E_{exact}|} 
\end{equation}
This provides an application agnostic accuracy measurement that will extends in this stacks future beyond quantum chemistry to any domain (future 
applications can include Optimization Problems, Financial Portfolios, etc.) where the stack can compute variational ground state energies.
